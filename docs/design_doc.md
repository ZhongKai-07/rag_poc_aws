  1. 概述

  1.1 项目背景

  本系统是一个企业级检索增强生成解决方案，旨在通过AWS云服务实现PDF文档的智能处理与问答。系统面向证券研究报告分析场景，支
  持金融报告的自动化处理、语义检索和智能问答。

  1.2 系统目标

  - 智能文档处理: 支持PDF文档的自动化解析、分块和向量化存储
  - 语义检索: 基于向量相似度的精准文档检索能力
  - 智能问答: 基于检索上下文生成准确、可溯源的答案
  - 多模态支持: 支持文档中文本和图像内容的综合处理

  1.3 技术选型

  ┌────────────┬──────────────────────────────┬──────────────────────────────┐
  │    层次    │           技术选型           │           选型理由           │
  ├────────────┼──────────────────────────────┼──────────────────────────────┤
  │ 前端框架   │ React 18 + TypeScript + Vite │ 现代化前端技术栈，开发效率高 │
  ├────────────┼──────────────────────────────┼──────────────────────────────┤
  │ UI组件库   │ Shadcn/ui + Tailwind CSS     │ 高度可定制，样式统一         │
  ├────────────┼──────────────────────────────┼──────────────────────────────┤
  │ 后端框架   │ FastAPI (Python)             │ 异步高性能，自动API文档      │
  ├────────────┼──────────────────────────────┼──────────────────────────────┤
  │ 向量数据库 │ AWS OpenSearch Service       │ 托管服务，原生支持kNN        │
  ├────────────┼──────────────────────────────┼──────────────────────────────┤
  │ 嵌入模型   │ AWS Bedrock Titan            │ 托管服务，1536维高质量向量   │

  2. 业务背景与场景

  2.1 业务背景

  本系统是企业内部知识库平台(Internal Knowledge Base for Ops)的核心智能问答引擎，旨在通过RAG技术提升运营人员的工作效率，降低人工查询制度文档的时间成本，同时支持业务决策的智能化辅助。

  2.2 核心业务场景

  2.2.1 场景一: OPS/COB合规问答

  **业务描述:**

  运营部门(Ops)日常需要频繁查询各类制度文档、合规要求、操作规程等内容。传统方式需要人工翻阅大量PDF文档，效率低下且容易遗漏关键信息。

  **核心需求:**
  - 快速检索: 支持自然语言提问，秒级返回相关答案
  - 精准溯源: 答案需标注来源文档和具体章节
  - 多文档关联: 一个问题可能涉及多份制度文档的交叉引用
  - 版本管理: 制度文档存在版本更新，需支持历史版本查询

  **典型问题示例:**
  ```
  Q: COB业务中，客户投诉处理的标准流程是什么？
  Q: 新版操作规程对清算时间有什么调整？
  Q: 合规检查的频率要求是多少？
  ```

  **技术挑战:**
  - 制度文档格式多样(PDF/Word/Excel)
  - 专业术语较多，需要精确匹配
  - 文档更新频繁，索引需要实时同步

  **RAG解决方案:**

  2.2.2 场景二: 抵押品智能筛选

  **业务描述:**

  在融资融券、股票质押等业务场景中，需要对抵押品进行合规性检查和筛选。传统方式依赖业务人员人工判断，效率低且存在漏判风险。

  **核心需求:**
  - 合规检查: 自动判断抵押品是否符合监管要求
  - 智能筛选: 根据业务需求推荐合适的抵押品
  - 风险评估: 结合抵押品历史数据进行风险提示
  - 规则匹配: 支持复杂业务规则的自动匹配

  **典型问题示例:**
  ```
  Q: 这只股票可以作为融资融券的担保品吗？
  Q: 符合以下条件的抵押品有哪些：市盈率<30、流动性>1000万、非ST股？
  Q: 该抵押品的风险等级是什么？
  ```

  **技术挑战:**
  - 需要结合结构化数据(股票行情)和非结构化数据(公告、研报)
  - 业务规则复杂，涉及多个条件的组合判断
  - 数据实时性要求高

  # 实现代码 (document_processing.py)

  def process_file(self, file_path, index_name: str):
      """处理PDF文件，优先使用Docling，失败则回退到PyPDF2"""

      if self.doc_converter and DOCLING_AVAILABLE:
          try:
              # 尝试使用Docling解析
              self._process_with_docling(file_path, index_name)
          except Exception as docling_error:
              # Docling失败，回退到PyPDF2
              logging.warning(f"Docling failed: {docling_error}. Falling back to PyPDF2")
              self._process_with_pypdf(file_path, index_name)
      else:
          # 直接使用PyPDF2
          self._process_with_pypdf(file_path, index_name)

  5.3 三种搜索模式实现

  # opensearch_search.py

  def similarity_search(..., search_method: str = "vector"):
      """
      三种搜索模式实现
      """

      if search_method == "vector":
          # 纯向量搜索
          query_vector = embeddings.embed_query(query)
          hits = vector_search(client, index_name, query_vector, vec_docs_num)

      elif search_method == "text":
          # 纯文本搜索
          hits = text_search(client, index_name, query, txt_docs_num)

      else:  # mix
          # 混合搜索：向量 + 文本
          query_vector = embeddings.embed_query(query)

          # 向量搜索
          vec_hits = vector_search(client, index_name, query_vector, vec_docs_num)
          vec_results = _format_results(vec_hits, ...)

          # 文本搜索
          text_hits = text_search(client, index_name, query, txt_docs_num)
          text_results = _format_results(text_hits, ...)

          # 合并去重
          all_results.extend(vec_results)
          all_results.extend(text_results)

          # 按分数排序，去重
          unique_results = _deduplicate_and_sort(all_results)

  def get_reranker_scores_bedrock(text_query, documents, modelId: str = None):
      """
      使用Bedrock Rerank模型对文档重新排序

      Args:
          text_query: 用户查询

              }
          }
      )

      return response['results']  # [{index, relevanceScore}, ...]

  9. 多租户与权限管理

  9.1 多租户需求分析

  本系统面向企业内部多个业务部门，需要支持数据隔离和权限控制。主要需求包括：

  | 需求维度 | 说明 |
  |----------|------|
  | 数据隔离 | 不同部门的数据需要逻辑隔离，避免跨部门数据泄露 |
  | 权限控制 | 基于角色的访问控制(RBAC)，区分管理员和普通用户权限 |
  | 审计追踪 | 记录用户的操作日志，支持合规审计 |
  | 灵活扩展 | 支持新增租户、权限调整等运营需求 |

  9.2 多租户架构方案

  本系统支持两种多租户隔离方案，根据业务场景和安全要求选择：

  9.2.1 方案一: 物理隔离 (Index per Tenant)

  **实现方式:**
  - 每个租户创建独立的OpenSearch索引
  - 索引命名规则: `{tenant_id}_{index_name}`
  - 通过索引级别的访问控制实现数据隔离

  **索引结构设计:**
  ```json
  // 租户A的索引: tenant_ScS    "mappings": {
      "properties": {
        "sentence_vector": {"type": "knn_vector", "dimension": 1536},
        "paragraph": {"type": "text"},
        "sentence": {"type": "text"},
        "metadata": {
          "properties": {
            "source": {"type": "keyword"},
            "created_at": {"type": "date"},
            "created_by": {"type": "keyword"}
          }
        }
      }
    }
  }
  ```

  **访问控制实现:**
  ```java
  // SpringBoot 服务层实现
  @Service
  public class DocumentService {

      public String getDocumentIndex(String tenantId, String documentId) {
          // 构建租户专属索引名
          String tenantIndex = tenantId + "_documents";

          // 仅查询该租户的索引
          return opensearchClient.get(index -> index
              .index(tenantIndex)
              .id(documentId)
          );
      }
  }
  ```

  **优势:**
  - 数据完全隔离，安全性最高
  - 支持索引级别的备份和恢复
  - 便于数据迁移和清理
  - 性能互不影响

  **劣势:**
  - 索引数量多，管理复杂
  - 资源占用较高
  - 跨租户查询困难

  **适用场景:**
  - 数据安全要求极高的业务场景
  - 租户数量较少(建议<50个)
  - 需要独立的备份恢复策略

  9.2.2 方案二: 逻辑隔离 (Metadata Filtering)

  **实现方式:**
  - 所有租户共享同一个索引
  - 每个文档添加`tenant_id`元数据字段
  - 查询时通过过滤条件实现数据隔离

  **索引结构设计:**
  ```json
  // 共享索引: shared_documents
  {
    "mappings": {
      "properties": {
        "sentence_vector": {"type": "knn_vector", "dimension": 1536},
        "paragraph": {"type": "text"},
        "sentence": {"type": "text"},
        "tenant_id": {"type": "keyword"},      // 租户标识
        "department": {"type": "keyword"},     // 部门标识
        "access_level": {"type": "keyword"},   // 访问级别
        "metadata": {
          "properties": {
            "source": {"type": "keyword"},
            "created_at": {"type": "date"},
            "created_by": {"type": "keyword"}
          }
        }
      }
    }
  }
  ```

  **查询过滤实现:**
  ```java
  // SpringBoot 服务层实现
  @Service
  public class DocumentService {

      public List<Document> searchDocuments(String tenantId, String query) {
          // 构建带租户过滤的查询
          SearchResponse<Document> response = opensearchClient.search(s -> s
              .index("shared_documents")
              .query(q -> q
                  .bool(b -> b
                      .must(m -> m
                          .knn(k -> k
                              .field("sentence_vector")
                              .vector(getEmbedding(query))
                              .k(10)
                          )
                      )
                      .filter(f -> f
                          .term(t -> t
                              .field("tenant_id")
                              .value(tenantId)
                          )
                      )
                  )
              ),
              Document.class
          );

          return response.hits().hits().stream()
              .map(Hit::source)
              .collect(Collectors.toList());
      }
  }
  ```

  **优势:**
  - 索引管理简单，资源利用率高
  - 支持跨租户聚合分析
  - 易于扩展新租户

  **劣势:**
  - 需要严格的过滤条件，有数据泄露风险
  - 查询性能受数据量影响
  - 备份恢复需要按租户过滤

  **适用场景:**
  - 租户数量多(>50个)
  - 数据安全要求适中
  - 需要跨租户分析能力

      @Override
      public boolean preHandle(HttpServletRequest request,
                               HttpServletResponse response,
                               Object handler) {
          // 1. 从JWT获取用户信息
          String token = request.getHeader("Authorization");
          UserInfo user = jwtService.parseToken(token);

          // 2. 设置租户上下文
          TenantContext.setCurrentTenant(user.getTenantId());

  5.1 双PDF解析器策略

  问题背景: Docling在Windows环境下存在资源文件路径问题

  解决方案: 主解析器+ 备用解析器（PyPDF2）

  # 实现代码 (document_processing.py)

  def process_file(self, file_path, index_name: str):
      """处理PDF文件，优先使用Docling，失败则回退到PyPDF2"""

      if self.doc_converter and DOCLING_AVAILABLE:
          try:
              # 尝试使用Docling解析
              self._process_with_docling(file_path, index_name)
          except Exception as docling_error:
              # Docling失败，回退到PyPDF2
              logging.warning(f"Docling failed: {docling_error}. Falling back to PyPDF2")
              self._process_with_pypdf(file_path, index_name)
      else:
          # 直接使用PyPDF2
          self._process_with_pypdf(file_path, index_name)

  5.3 三种搜索模式实现

  # opensearch_search.py

  def similarity_search(..., search_method: str = "vector"):
      """
      三种搜索模式实现
      """

      if search_method == "vector":
          # 纯向量搜索
          query_vector = embeddings.embed_query(query)
          hits = vector_search(client, index_name, query_vector, vec_docs_num)

      elif search_method == "text":
          # 纯文本搜索
          hits = text_search(client, index_name, query, txt_docs_num)

      else:  # mix
          # 混合搜索：向量 + 文本
          query_vector = embeddings.embed_query(query)

          # 向量搜索
          vec_hits = vector_search(client, index_name, query_vector, vec_docs_num)
          vec_results = _format_results(vec_hits, ...)

          # 文本搜索
          text_hits = text_search(client, index_name, query, txt_docs_num)
          text_results = _format_results(text_hits, ...)

          # 合并去重
          all_results.extend(vec_results)
          all_results.extend(text_results)

          # 按分数排序，去重
          unique_results = _deduplicate_and_sort(all_results)

  9. 多租户与权限管理

  9.1 多租户需求分析

  本系统面向企业内部多个业务部门，需要支持数据隔离和权限控制。主要需求包括：

  | 需求维度 | 说明 |
  |----------|------|
  | 数据隔离 | 不同部门的数据需要逻辑隔离，避免跨部门数据泄露 |
  | 权限控制 | 基于角色的访问控制(RBAC)，区分管理员和普通用户权限 |
  | 审计追踪 | 记录用户的操作日志，支持合规审计 |
  | 灵活扩展 | 支持新增租户、权限调整等运营需求 |

  9.2 多租户架构方案

  本系统支持两种多租户隔离方案，根据业务场景和安全要求选择：

  9.2.1 方案一: 物理隔离 (Index per Tenant)

  **实现方式:**
  - 每个租户创建独立的OpenSearch索引
  - 索引命名规则: `{tenant_id}_{index_name}`
  - 通过索引级别的访问控制实现数据隔离

  **索引结构设计:**
  ```json
  // 租户A的索引: tenant_a_documents
  {
    "mappings": {
      "properties": {
        "sentence_vector": {"type": "knn_vector", "dimension": 1536},
        "paragraph": {"type": "text"},
        "sentence": {"type": "text"},
        "metadata": {
          "properties": {
            "source": {"type": "keyword"},
            "created_at": {"type": "date"},
            "created_by": {"type": "keyword"}
          }
        }
      }
    }
  }
  ```

  **访问控制实现:**
  ```java
  // SpringBoot 服务层实现
  @Service
  public class DocumentService {

      public String getDocumentIndex(String tenantId, String documentId) {
          // 构建租户专属索引名
          String tenantIndex = tenantId + "_documents";

          // 仅查询该租户的索引
          return opensearchClient.get(index -> index
              .index(tenantIndex)
              .id(documentId)
          );
      }
  }
  ```

  **优势:**
  - 数据完全隔离，安全性最高
  - 支持索引级别的备份和恢复
  - 便于数据迁移和清理
  - 性能互不影响

  **劣势:**
  - 索引数量多，管理复杂
  - 资源占用较高
  - 跨租户查询困难

  **适用场景:**
  - 数据安全要求极高的业务场景
  - 租户数量较少(建议<50个)
  - 需要独立的备份恢复策略

  9.2.2 方案二: 逻辑隔离 (Metadata Filtering)

  **实现方式:**
  - 所有租户共享同一个索引
  - 每个文档添加`tenant_id`元数据字段
  - 查询时通过过滤条件实现数据隔离

  **索引结构设计:**
  ```json
  // 共享索引: shared_documents
  {
    "mappings": {
      "properties": {
        "sentence_vector": {"type": "knn_vector", "dimension": 1536},
        "paragraph": {"type": "text"},
        "sentence": {"type": "text"},
        "tenant_id": {"type": "keyword"},      // 租户标识
        "department": {"type": "keyword"},     // 部门标识
        "access_level": {"type": "keyword"},   // 访问级别
        "metadata": {
          "properties": {
            "source": {"type": "keyword"},
            "created_at": {"type": "date"},
            "created_by": {"type": "keyword"}
          }
        }
      }
    }
  }
  ```

  **查询过滤实现:**
  ```java
  // SpringBoot 服务层实现
  @Service
  public class DocumentService {

      public List<Document> searchDocuments(String tenantId, String query) {
          // 构建带租户过滤的查询
          SearchResponse<Document> response = opensearchClient.search(s -> s
              .index("shared_documents")
              .query(q -> q
                  .bool(b -> b
                      .must(m -> m
                          .knn(k -> k
                              .field("sentence_vector")
                              .vector(getEmbedding(query))
                              .k(10)
                          )
                      )
                      .filter(f -> f
                          .term(t -> t
                              .field("tenant_id")
                              .value(tenantId)
                          )
                      )
                  )
              ),
              Document.class
          );

          return response.hits().hits().stream()
              .map(Hit::source)
              .collect(Collectors.toList());
      }
  }
  ```

  **优势:**
  - 索引管理简单，资源利用率高
  - 支持跨租户聚合分析
  - 易于扩展新租户

  **劣势:**
  - 需要严格的过滤条件，有数据泄露风险
  - 查询性能受数据量影响
  - 备份恢复需要按租户过滤

  **适用场景:**
  - 租户数量多(>50个)
  - 数据安全要求适中
  - 需要跨租户分析能力

  **SpringBoot权限拦截器实现:**
  ```java
  @Component
  public class TenantInterceptor implements HandlerInterceptor {

      @Override
      public boolean preHandle(HttpServletRequest request,
                               HttpServletResponse response,
                               Object handler) {
          // 1. 从JWT获取用户信息
          String token = request.getHeader("Authorization");
          UserInfo user = jwtService.parseToken(token);

          // 2. 设置租户上下文
          TenantContext.setCurrentTenant(user.getTenantId());

          // 3. 权限校验
          if (!permissionService.hasPermission(user, request.getRequestURI())) {
              response.setStatus(403);
              return false;
          }

          return true;
      }
  }
  ```

  9.4 OpenSearch安全配置

  9.4.1 安全插件配置

  OpenSearch支持通过安全插件(Security Plugin)实现细粒度的访问控制:

  **角色映射配置:**
  ```yaml
  # opensearch-security/roles_mapping.yml
  tenant_a_admin:
    users:
      - "tenant_a_admin_user"
    backend_roles:
      - "tenant_a_admin_group"

  tenant_a_user:
    users:
      - "tenant_a_user1"
      - "tenant_a_user2"
    backend_roles:
      - "tenant_a_group"
  ```

  **索引权限配置:**
  ```yaml
  # opensearch-security/roles.yml
  tenant_a_admin:
    index_permissions:
      - index_patterns:
          - "tenant_a_*"
        allowed_actions:
          - "indices:data/read/*"
          - "indices:data/write/*"
          - "indices:admin/create"
          - "indices:admin/delete"

  tenant_a_user:
    index_permissions:
      - index_patterns:
          - "tenant_a_documents"
        allowed_actions:
          - "indices:data/read/*"
  ```
